                                                                                          Error messages

                                                                                          Error messages provide immediate feedback when something goes wrong —a user-triggered error, an unexpected event, or a technical problem — that blocks use of our apps. The goal of an error message is to give the user info and control at a time when they may feel frustrated. Specifically, an error message should:

                                                                                          • Say what's happening or indicate what’s wrong

                                                                                          • Explain why it's happening

                                                                                          • Tell the user what they can do to fix the problem or move forward


                                                                                          TL;DR

                                                                                          • Good error messages include a concise statement of what went wrong, a recommendation to fix the error, and a clear affordance to take action

                                                                                          • Avoid sounding overly negative, since users are already frustrated. Go for optimism instead.

                                                                                          • Never blame the user, even if an error is their fault

                                                                                          • Don’t use Something went wrong; be more specific and therefore reassuring

                                                                                          • Always provide a way forward, even if it’s just to say Try again

                                                                                          • Avoid apologizing unless Uber is at fault for the issue

                                                                                          Content principles

                                                                                          A good framework for writing error messages grounds them in a bit of empathy. We always want to be on the user's side.

                                                                                          Optimistic

                                                                                          Error messages should be optimistic, but not casual and breezy, to give users confidence that they can resolve the issue with our help, or that we’re addressing the issue (if it’s technical).

                                                                                          Realistic

                                                                                          Don’t downplay issues or overstate them, and avoid sugarcoating the error message with flowery language or promises we can’t keep. Reinforce trust with text that’s short and jargon-free. 

                                                                                          Supportive

                                                                                          Never blame the user for the error, even if it’s their fault. And avoid overuse of sorry. Rather, keep the content positive and focused on helping the user move forward.


                                                                                          Types of errors

                                                                                          Errors can be categorized into 3 categories. For all of these and depending on context, the corresponding error message can appear inline (presented directly in the UI or below a text field), or in a dialog or other dedicated surface.

                                                                                          User-triggered error

                                                                                          These errors happen when the user enters something wrong or invalid in a traditional form field or any other input field, or fails to enter content in a required field or to make a required choice.

                                                                                          An unexpected event

                                                                                          This is when an external factor intervenes to interrupt the happy path, like heavy traffic or a cancellation (driver, rider, restaurant).

                                                                                          Technical problem

                                                                                          Technical issues often include server outages, connection issues, platform issues, and so on.



                                                                                          Writing guidelines

                                                                                          Getting the tone right

                                                                                          Term

                                                                                          Guidance

                                                                                          Example

                                                                                          Can't

                                                                                          OK

                                                                                          Use can’t when a user isn’t allowed to do something in the app (such as change their destination during a trip because they don’t have enough Uber Cash to cover the new price). Can’t suggests it’s not possible. Note: For inclusivity, don't use terms like able and unable.

                                                                                          Couldn't

                                                                                          OK

                                                                                          Use couldn’t rather than failed when an action didn't work, possibly for an unknown reason (for instance, Couldn’t save your place). This suggests it didn’t work this time; trying again might work.

                                                                                          Do X

                                                                                          Avoid

                                                                                          Try not to be directive. It could sound bossy.


                                                                                          Don’t: Do X

                                                                                          Do: Try doing X 

                                                                                          Error

                                                                                          Avoid

                                                                                          Don't use, especially in messages that are user-triggered. Try issue instead.

                                                                                          Failed

                                                                                          Avoid

                                                                                          This sounds overly negative.


                                                                                          Don’t: Your credit card failed

                                                                                          Do: Couldn’t process your payment

                                                                                          Don't: GPS failed

                                                                                          Do: No GPS; GPS not available

                                                                                          Incorrect

                                                                                          Avoid

                                                                                          This is long and sounds formal. Use wrong, or find other softer wording.

                                                                                          Invalid

                                                                                          Avoid

                                                                                          There’s usually something more specific and useful to say than to declare something invalid. Better to say why it’s invalid.

                                                                                          Don’t: Invalid password

                                                                                          Do: Your password must have 8 characters

                                                                                          Issue

                                                                                          OK

                                                                                          This is a better, more neutral word to use than error.

                                                                                          Oops

                                                                                          Never use

                                                                                          It sets the wrong tone and trivializes issues that matter to the user.

                                                                                          Please

                                                                                          OK

                                                                                          Use when you want to soften the tone of an instruction; don’t overuse. Refer to Product tone for more guidance.

                                                                                          Something went wrong

                                                                                          Avoid

                                                                                          Find another way to be noncommittal but less vague.

                                                                                          Don’t: Something went wrong

                                                                                          Do: There was a technical issue

                                                                                          Sorry

                                                                                          OK

                                                                                          Apologize when we have something to be sorry for, like a technical issue that’s our fault. It’s sometimes OK to soften the tone of a message that might be particularly frustrating. Refer to Product tone for more guidance.

                                                                                          Don’t: I’m sorry

                                                                                          Do: Sorry for the inconvenience

                                                                                          Wrong

                                                                                          OK

                                                                                          This is a better word choice than incorrect, which sounds formal. But try to find a better way to word the message.

                                                                                          Inline error messages

                                                                                          Inline error messages include things like form validation errors and issues with payment methods. They’re presented in situ, rather than appearing in a dialog or other surface.

                                                                                          Use body text only: Text should be a short phrase or sentence that simply labels the issue or provides a brief instruction. Don't use end punctuation:

                                                                                          • Card expired

                                                                                          • Enter a 9-digit SSN

                                                                                          • The passwords don’t match 

                                                                                          • A password must contain at least one special character

                                                                                          Dialog error messages

                                                                                          Dialog error messages require a header, body text, and a CTA.

                                                                                          Header

                                                                                          • Use a concise, one-line statement of what went wrong or what’s already happening to address the issue

                                                                                          • Usually a short phrase; rarely a sentence

                                                                                          Body text

                                                                                          • Brief information (1 or 2 short sentences) with more context or detail about the error — but only if it might not be immediately obvious

                                                                                          • Say what the user can do to recover, or simply tell them to try again

                                                                                          • Don’t repeat the header

                                                                                          • For technical issues, include expectations about impact and duration, if possible

                                                                                          CTA

                                                                                          Provide a clear affordance for the user to take action, if possible, or to simply acknowledge the message.

                                                                                          • Stick to OK as a simple acknowledgment that closes the dialog.

                                                                                          • Don't use Got it because it's tonally off in a situation where the user is likely frustrated.

                                                                                          Header

                                                                                          Wrong ID

                                                                                          • You entered the wrong ID

                                                                                          • Invalid ID

                                                                                          • Use the simplest words. Wrong is better than incorrect

                                                                                          • Avoid using invalid for info that has been entered incorrectly

                                                                                          Try entering it again. You can find the ID on the scooter handlebar.

                                                                                          Try to include details that help the user recover.

                                                                                          CTA

                                                                                          OK

                                                                                          Got it

                                                                                          Got it is a bit too glib when you’re asking the user to acknowledge an error they made.

                                                                                          • Lead with the good news, if possible

                                                                                          • Avoid sounding too negative

                                                                                          Body

                                                                                          Your driver canceled, but you’ll be connected with another one shortly.

                                                                                          • Don’t repeat the header

                                                                                          • Provide helpful detail about the resolution, or provide a way to resolve the issue

                                                                                          Sounds good

                                                                                          • Avoid making it sound like we don’t know what’s going on

                                                                                          It looks like the restaurant is no longer in business. Sorry for the confusion, Tom. We have other places nearby you might want to try.

                                                                                          Check out these other great places near you!


                                                                                          Avoid sounding enthusiastic in the face of a user’s frustration.

                                                                                          OK

                                                                                          Close

                                                                                          Don't

                                                                                          Header

                                                                                          We couldn’t connect with the bank

                                                                                          • Something went wrong

                                                                                          • Oops, there was an issue with your bank

                                                                                          • Be definitive about the issue

                                                                                          • Avoid making it sound like we don’t know what’s going on

                                                                                          • The server is down. Try again later.

                                                                                          • Avoid using technical terms
